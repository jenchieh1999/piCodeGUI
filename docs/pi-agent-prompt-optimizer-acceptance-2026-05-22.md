# Pi Agent Desktop 提示词优化功能验收报告

日期：2026-05-22

## 1. 验收结论

当前提示词优化功能已完成第一轮端到端自测，可进入人工验收阶段。

本轮重点验收：

- 输入框提示词优化预览卡片。
- 优化结果接受、复制、重新优化、关闭和撤销链路。
- 设置页单独选择“提示词优化模型”。
- 后端 `preferredOptimizerModel` 优先级。
- 后端本地 fallback、模式识别、上下文保留和质量元数据。
- 默认超时从 `1s` 调整为 `6s`，并支持 `PI_AGENT_PROMPT_OPTIMIZER_TIMEOUT_MS=1000-30000`。
- JSON 响应头补充 `charset=utf-8`，避免部分客户端读取中文时出现乱码。

## 4. 人工验收总步骤

建议先准备：

1. 打开 Pi Agent Desktop。
2. 进入 `设置 -> 模型`，指定一个快速模型作为“提示词优化模型”，例如 Mini Hub GPT、GLM Air、Flash、Haiku 等。
3. 回到主对话输入框。
4. 每个样例执行一次“优化提示词”，观察预览卡片。
5. 分别测试“接受”“复制”“重新优化”“关闭”“Ctrl/Cmd+Z 撤销”。

通用验收标准：

- 优化前内容不会被立即覆盖。
- 预览卡片能展示优化前、优化后、模式、来源、评分、风险。
- 接受后才替换输入框。
- 复制按钮能复制优化后的提示词。
- 关闭预览不会改变输入框。
- 接受后能撤销回优化前内容。
- 中文不乱码。
- 文件路径、错误信息、版本号、环境变量、命令不应丢失。

## 5. 验收测试样例

### 5.1 普通模糊需求优化

输入：

```text
帮我看看这个页面有点怪，优化一下
```

建议模式：`智能` 或 `UI`

期望：

- 优化后应明确页面范围、具体问题、视觉/交互/响应式要求。
- 应要求实现后进行检查或截图验收。
- 不应变成泛泛的设计建议。

### 5.2 UI 风格优化

输入：

```text
把凭据页面改得更苹果风格一点，现在太复杂了，输入框和按钮也不好看
```

建议模式：`UI`

期望：

- 输出包含信息架构、视觉层级、表单状态、按钮布局、空状态/错误状态。
- 保留“苹果风格”“凭据页面”等关键意图。
- 不应只说“美化 UI”，要能让 coding agent 直接执行。

### 5.3 Debug 问题优化

输入：

```text
我接了智谱 glm-5.1，但是还是没用，配置了 endpoint 和 key，发送消息失败
```

建议模式：`调试`

期望：

- 输出应包含复现条件、检查项、日志/API 响应、模型名大小写、endpoint、key、provider 映射。
- 应保留 `glm-5.1`、`endpoint`、`key`。
- 不应直接给结论说“一定是 key 错了”。

### 5.4 代码执行任务优化

输入：

```text
文件区的右键复制粘贴剪切撤销还不完整，帮我补齐
```

建议模式：`执行`

期望：

- 输出应包含功能范围、快捷键、右键菜单、文件移动/复制/撤销行为、验证方式。
- 应提示不要破坏已有文件区拖拽和预览逻辑。
- 应有可验收清单。

### 5.5 代码审查任务优化

输入：

```text
帮我检查一下扩展和包市场有没有阻塞问题
```

建议模式：`审查`

期望：

- 输出应要求先审查代码和运行路径，再列阻塞性问题。
- 关注黑屏、无法安装、API 错误、状态同步、空状态。
- 输出格式应偏 code review：发现、影响、修复建议、验证。

### 5.6 研究/调研任务优化

输入：

```text
分析一下 Electron 过渡壳是否需要迁移到 Tauri
```

建议模式：`研究`

期望：

- 输出应要求对比性能、包体、生态、插件、发布链路、风险、迁移成本。
- 应要求区分事实、推断和建议。
- 需要明确最终输出结构，如结论、对比表、迁移路线。

### 5.7 智能体聊天室任务优化

输入：

```text
我需要让多个 agent 分成两个观点组互相辩论，最后由中立组总结
```

建议模式：`智能体`

期望：

- 输出应包含议题、视角 A、视角 B、子 Agent、证据、辩论轮次、中立总结、验收标准。
- 不应出现带政治色彩的“左派/右派”默认文案。
- 应强调可改名的视角组和中立评审。

### 5.8 文件引用保留

输入：

```text
请根据 @docs/pi-agent-agents-chatroom-design-plan.md 和 @frontend/src/components/agents-room/AgentsRoomView.tsx 优化智能体聊天室阅读体验，右侧证据板看不全
```

建议模式：`执行` 或 `UI`

期望：

- 优化后必须保留两个 `@文件路径`。
- 应明确右侧证据板、中立输出、Markdown 阅读体验、滚动区域。
- 不应把文件路径改写成自然语言而丢失引用。

### 5.9 选区优化

输入整段：

```text
请你帮我实现这个功能。
要求：用户可以在文件区复制和粘贴文件。
另外终端也不要退出。
```

操作：

1. 只选中第一句“请你帮我实现这个功能。”
2. 点击优化提示词。
3. 接受预览。

期望：

- 预览卡片“优化前”只显示选区文本。
- 接受后只替换第一句。
- 后两句要求保持原样。

### 5.10 多文件 + Git 上下文

输入：

```text
请帮我检查当前改动，看看有没有安全问题，尤其是网络请求、凭据存储、自动更新和插件安装
```

建议模式：`审查`

期望：

- 输出应要求检查 git diff、敏感信息、网络边界、权限、插件安装来源。
- 应包含“需要用户处理的手工步骤”。
- 不应只做普通代码格式检查。

### 5.11 图片/截图场景

输入：

```text
红框里的按钮位置不对，帮我优化成苹果风格
```

操作：

1. 先在输入框附加一张截图。
2. 再点击优化提示词。

期望：

- 优化后应保留“参考截图/红框位置”的语义。
- 应要求根据截图定位问题，而不是凭空猜测。
- 输出应包含视觉对齐、间距、层级、响应式、验收截图。

### 5.12 超时 fallback 验收

输入：

```text
请把这段需求优化成非常清晰的开发任务：智能体聊天室需要支持 quick/deep 模型选择、证据筛选、复制证据、打开来源文件
```

操作：

1. 临时把快速模型配置成不可用，或设置很短超时：

```powershell
$env:PI_AGENT_PROMPT_OPTIMIZER_TIMEOUT_MS="1000"
```

2. 点击优化提示词。

期望：

- 不应长时间卡住。
- 应 fallback 到本地模板。
- 预览卡片或提示中能看到本地 fallback 来源。
- 优化结果仍可用，不乱码。

### 5.13 版本号和命令保留

输入：

```text
请重新出包，版本为 0.1.2，以后打包版本只打双架构包合包，命令可以参考 npm.cmd run build
```

建议模式：`执行`

期望：

- 必须保留 `0.1.2`。
- 必须保留 `npm.cmd run build`。
- 应明确“只打双架构合包”的约束。
- 不应改成泛泛的“更新版本并构建”。

### 5.14 智谱模型配置场景

输入：

```text
配置智谱模型时，需要用小写的 glm，我输入了 GLM-5.1 之后没有生效
```

建议模式：`调试`

期望：

- 优化后应保留 `glm` 小写要求。
- 应包含 provider/modelId 规范化、配置页显示、请求发送、日志验证。
- 不应把 `glm-5.1` 改成 `GLM-5.1`。

## 6. 模式差异验收

使用同一句输入分别选择不同模式：

```text
帮我优化智能体聊天室，现在证据板不好用，模型也不听设置
```

期望差异：

| 模式 | 期望输出重点 |
| --- | --- |
| 智能 | 自动识别为产品 + 代码任务，输出综合任务 |
| 执行 | 明确实现步骤、文件范围、验证命令 |
| 调试 | 关注模型设置为什么未生效、数据流和日志 |
| 审查 | 要求检查已开发功能的问题和阻塞风险 |
| UI | 关注证据板布局、筛选、阅读体验、按钮位置 |
| 智能体 | 输出多视角讨论结构和综合评审标准 |

## 7. 通过标准

一轮人工验收可视为通过，当满足：

- 至少 8 个样例通过。
- `5.8 文件引用保留`、`5.9 选区优化`、`5.12 超时 fallback`、`5.14 智谱模型配置场景` 必须通过。
- 所有中文输出无乱码。
- 预览卡片不会在用户接受前改写输入框。
- 接受后可以撤销。

## 8. 已知剩余风险

- 真模型链路依赖用户本机 API Key、endpoint、网络状态，仍需在真实配置下人工确认。
- `prompt-engineering-expert` skill 已有显式启用入口，但还没有和 Skill Hub / Trust Center 完整 UI 联动。
- 当前预览是双栏轻量对比，不是逐词或逐行 diff。
- 仍缺少正式单元测试和快照测试。
